### 2.4 年费 vs 一次性开通 · 对比分析

#### 两种模式的本质差异

| 维度 | 年费制 | 一次性买断 |
|------|--------|------------|
| 用户心理 | "每年都在用，每年付" | "买了就是我的，永久用" |
| 付费门槛 | 低（¥49起） | 高（通常¥199起） |
| 产品方收入 | 可持续，支撑长期维护 | 一次性，需持续拉新 |
| 产品更新 | 年费支撑持续更新产品库 | 无动力更新旧版本 |
| 用户预期 | 持续获得新功能+新数据 | 买断后无持续服务预期 |
| 适合场景 | 数据持续变化(产品库/费率) | 功能固化型工具 |

#### 我们的推荐策略：年费为主，买断为辅

| 版本 | 年费 | 一次性买断 | 建议 |
|------|------|-----------|------|
| 守护版 | ¥49/年 | ¥99 永久 | 年费优先（¥49门槛极低） |
| 标准版 | ¥99/年 | ¥199 永久 | 年费优先（送通知+报告已超值） |
| 大家庭版 | ¥168/年 | ¥299 永久 | 年费优先（产品库更新依赖年费） |
| 管家版 | ¥399/年 | **不开放买断** | 仅年费（服务深度决定） |

> 核心判断：保险产品库的费率、停售、新品需要持续更新。买断版不享受产品库更新（停留在购买时的版本），年费版持续获取最新数据。这个差异本身就引导用户选年费。买断版存在的主要价值是**降低用户决策的心理障碍**——"虽然我推荐年费，但如果你想一次付清也可以"——这种对比让年费显得更划算。

#### 消费心理学分析

```
场景1：用户看到 ¥49/年 vs ¥99永久
  → 计算：2年就回本
  → 但 ¥49 低到不需要计算，冲动即可支付
  → 结论：¥49 年费更容易转化

场景2：用户看到 ¥168/年 vs ¥299永久
  → 计算：不到2年回本
  → 但"168"吉利数字 + 产品库持续更新 = 年费更有价值感
  → 结论：年费更有竞争力

场景3：用户看到 ¥399/年，仅年费
  → 没有买断选项 → 不会触发"划不划算"的计算
  → 决策简化为："我需不需要这个服务？"
  → 结论：减少决策噪音
```

### 2.5 保险观念调查（个性化推荐的基础）

> 用户录入家庭成员后，可选择填写保险观念调查（可跳过）。调查结果用于：① 调整产品推荐权重 ② 判断用户认知水平以调整报告深度 ③ 积累用户画像数据以优化产品库合作。

#### 调查问卷设计（5个维度，6道题，约1分钟）

**维度1：了解度**
```
Q: 您对保险的了解程度？
  ○ 不太了解，需要专业建议
  ○ 知道基本险种（百万医疗/重疾/意外）
  ○ 比较熟悉，能自己对比产品
  ○ 非常专业，从事保险相关行业
```

**维度2：预算预期**
```
Q: 您计划为家庭保险配置的年预算范围？
  ○ ¥2,000以内（基础保障）
  ○ ¥2,000-5,000（标准配置）
  ○ ¥5,000-15,000（全面配置）
  ○ ¥15,000以上（顶配/含理财型）
  ○ 暂无明确预算，看推荐再定
```

**维度3：关注方向（多选）**
```
Q: 您最关注的保障方向？
  □ 重大疾病保障
  □ 意外伤害保障
  □ 住院/大病医疗报销
  □ 家庭经济支柱身故保障
  □ 子女教育/婚嫁规划
  □ 养老规划
  □ 财产/房屋保障
```

**维度4：保险公司偏好**
```
Q: 您对保险公司有偏好吗？
  ○ 无所谓，产品好就行
  ○ 偏好大品牌（人保/平安/太保/国寿）
  ○ 偏好性价比（中小公司也可以）
  ○ 偏好外资/合资公司
```

**维度5：保障内容侧重**
```
Q: 选择保险时您最看重什么？（排序1-3）
  [  ] 价格/性价比
  [  ] 保障范围/额度
  [  ] 续保条件/稳定性
  [  ] 公司品牌/偿付能力
  [  ] 健告宽松/投保容易
  [  ] 理赔口碑/服务体验
```

#### 调查结果如何影响推荐

```
原始引擎评分权重：价格×0.3 + 健告×0.2 + 续保×0.25 + 公司×0.25

用户填写调查后 → 动态调整权重：

示例1：用户选"偏好大品牌" + "看重公司品牌"
  → 调整：价格×0.15 + 健告×0.15 + 续保×0.2 + 公司×0.4 + 品牌×0.1
  → 结果：太保B款(AA)排名提升，众安(B类)排名下降

示例2：用户选"看重性价比" + "预算¥2,000以内"
  → 调整：价格×0.5 + 健告×0.15 + 续保×0.2 + 公司×0.15
  → 结果：达尔文12号(低价)排名提升，完美人生8号排名下降

示例3：用户选"关注重疾保障" + "看重保障范围"
  → 调整：价格×0.2 + 健告×0.15 + 续保×0.15 + 保障倍数×0.5
  → 结果：保额高/多次赔付的产品排名提升

示例4：用户跳过调查（默认）
  → 使用默认权重，覆盖大部分场景
```

#### 调查数据对产品库合作的价值

```
积累1000份调查数据后：
  → 发现60%用户预算在¥2,000-5,000
  → 发现70%用户最关注重疾保障
  → 产品库重点补充¥2,000-5,000区间的重疾险产品
  → 与保险公司合作时可提供用户需求数据支撑
  → 推荐精准度持续提升
```

## 三、权限与可见性模型

### 3.1 核心概念

```
用户 (User)          微信账号，唯一标识 = openid
家庭 (Family)        一个保障单元（如"林永聪家庭"、"林于雅家庭"）
成员资格 (Membership) 用户与家庭的关系：申请人 / 管理员 / 普通成员
```

### 3.2 可见性规则（铁律）

```
规则1：同一家庭内，所有成员可互看该家庭的全部保单数据
规则2：不同家庭之间，数据完全隔离（除非有共享成员）
规则3：如果一个用户同时属于家庭B和家庭C：
       → 该用户在B中看到B的数据
       → 该用户在C中看到C的数据
       → B的其他成员看不到C
       → C的其他成员看不到B
规则4：数据归属权归家庭申请人（创建者），申请人可删除家庭
规则5：用户可主动退出家庭，退出后不再看到该家庭数据
```

### 3.3 角色定义

| 角色 | 权限 | 说明 |
|------|------|------|
| **申请人** (Creator) | 全部权限 | 创建家庭、添加/移除成员、编辑所有保单、删除家庭、邀请新成员 |
| **管理员** (Admin) | 管理权限 | 编辑所有保单、邀请成员（不能删除家庭、不能移除创建者） |
| **成员** (Member) | 查看+编辑自己的保单 | 查看家庭全部保单、增删改自己录入的保单 |
| **只读成员** (Viewer) | 仅查看 | 只能看，不能改（适合给长辈看、给顾问看） |

### 3.4 场景示例

```
林永聪(A) 创建了：
  - 家庭B（林家·聪/欣/老妈/雅/嘉/微/老爸/权）— A是申请人
  - 家庭C（马家·岳父/岳母）— A是申请人

马伟雄(C) 创建了：
  - 家庭D（马家扩展·含其他亲戚）

可见性：
  A 登录 → 看到 [家庭B] 和 [家庭C] 的切换入口
  A 进入家庭B → 看到 聪/欣/老妈/雅/嘉/微/老爸/权 的全部保单
  A 进入家庭C → 看到 岳父/岳母 的全部保单
  
  C 登录 → 看到 [家庭C]（A创建的）和 [家庭D]（自己创建的）
  C 进入家庭C → 看到 岳父/岳母（共享家庭，A和C都能看到）
  C 进入家庭D → 看到D的数据

  A ❌ 看不到 家庭D
  C ❌ 看不到 家庭B
  A 和 C 在家庭C内可以看到彼此（共享家庭）
```

### 3.5 邀请流程

```
1. 申请人 在家庭设置页 点击「邀请成员」
2. 选择微信好友 / 生成邀请链接/小程序码
3. 被邀请人 点击链接 → 授权微信登录 → 自动加入家庭
4. 默认角色为「成员」，申请人可后续提升为「管理员」
5. 邀请链接有效期 48 小时，过期需重新生成
```

## 四、数据库设计（云开发）

### 4.1 集合设计

```
users（用户表）
  ├── _id: string (openid)
  ├── nickName: string
  ├── avatarUrl: string
  ├── phoneNumber: string (可选)
  ├── createdAt: datetime
  └── subscription: { tier: "free|standard|family|premium", expiresAt: datetime }

families（家庭表）
  ├── _id: string
  ├── name: string ("林家保险")
  ├── creatorId: string (→ users._id)
  ├── createdAt: datetime
  └── memberCount: number

memberships（成员资格表）
  ├── _id: string
  ├── userId: string (→ users._id)
  ├── familyId: string (→ families._id)
  ├── role: "creator" | "admin" | "member" | "viewer"
  ├── joinedAt: datetime
  └── invitedBy: string (→ users._id)

familyMembers（家庭成员表）
  ├── _id: string
  ├── familyId: string (→ families._id)
  ├── name: string
  ├── age: number
  ├── role: string
  ├── 医保: string
  ├── 负债: string
  ├── notes: string
  └── createdBy: string (→ users._id)

policies（保单表）
  ├── _id: string
  ├── familyId: string
  ├── memberId: string (→ familyMembers._id)
  ├── type/name/company/premium/coverage/guaranteedYears/...
  ├── screenshot: string (云存储fileID)
  ├── createdBy: string
  └── updatedAt: datetime

invitations（邀请表）
  ├── _id: string
  ├── familyId: string
  ├── inviterId: string
  ├── code: string (唯一邀请码)
  ├── role: "admin|member|viewer"
  ├── expiresAt: datetime
  └── acceptedBy: string (被邀请人userId，接受后填充)

reminders（提醒表）— 后期实现
  ├── _id: string
  ├── userId: string
  ├── policyId: string
  ├── type: "renewal" | "birthday_window" | "age_window"
  ├── remindAt: datetime
  ├── message: string
  └── status: "pending" | "sent" | "dismissed"
```

### 4.2 安全规则（云开发数据库权限）

```javascript
// memberships 读权限：仅当 userId == 当前用户的 openid
// families 读权限：当前用户的 memberships 中包含该 familyId
// familyMembers 读权限：当前用户属于该 member 所属的 family
// policies 读权限：当前用户属于该 policy 所属的 family

// 写权限：creator/admin 可写所有；member 只能写自己创建的；viewer 不能写
```
